# PWA + Deployment Plan

Turn `@opendaw/app-studio` into an installable, offline-capable PWA with a **consent-gated** update flow. Brief, issue-focused.

## Current facts (must integrate with)

- **Vite build**, output `dist/`, every chunk/asset filename carries the build `uuid`: `[name].{uuid}.js|css|ttf`.
- **CI base path** is per-release: `/{main|dev}/releases/{uuid}/`. Each deploy is an **immutable directory**. Apache `.htaccess` at web root rewrites the hostname to the current release dir; SPA fallback to `index.html`.
- `index.html` + `build-info.json` are served **no-cache**; uuid-named files are `immutable, max-age=1y`. Brotli `.br` served by content negotiation.
- **Update detection already exists** (`boot.ts:149`): polls `build-info.json?v=…` every 5s, compares `uuid`, gated on `navigator.onLine`, shows a passive `UpdateMessage` banner. Also a cache-mismatch guard (`boot.ts:132`) → `Dialogs.cache()`.
- **Cross-origin isolation is mandatory** (COOP `same-origin` / COEP `require-corp`) for AudioWorklet + SharedArrayBuffer. Boot aborts if `!crossOriginIsolated`.
- **Runtime assets**: `workers-main.js`, `processors.js` (~1.1MB worklet), `offline-engine.js` (loaded via `?url`); lazy `monaco-editor`, `@ffmpeg/ffmpeg` (WASM), `onnxruntime-web`; fonts in `public/fonts`.
- **Cloud assets** (network-only): samples/soundfonts from `assets.opendaw.studio` (streamed into OPFS), `api.opendaw.studio` (lists, publish, stats, latency), Dropbox/Google Drive auth, GitHub/sponsors. Local user data lives in **OPFS** (works offline already).
- Desktop-only (mobile is blocked at `main.ts`).

## Core design decisions

### 1. SW scope vs. release-directory layout (the hard one)
The SW must control `/` (stable URL) but assets live under `/{env}/releases/{uuid}/`. Two viable options:

- **A — root SW, no precache, runtime-cache only (recommended).** Serve `/sw.js` from web root (no-cache header). It does **not** ship a static precache manifest; it caches with a `StaleWhileRevalidate`/`CacheFirst` runtime strategy keyed by the immutable uuid filenames it observes being fetched. Survives the release-dir indirection without the SW needing to know the current uuid. Downside: full offline only after the relevant files have been visited once.
- **B — per-release SW with injected manifest.** `sw.{uuid}.js` lives in the release dir; root rewrites `/sw.js` → current release SW. Precise precache list, true first-load offline. Downside: the rewrite/scope dance is fragile, and `Service-Worker-Allowed` header is required to widen scope above the release dir.

**Recommendation: A.** It matches the immutable-uuid model, keeps the SW tiny, and makes the install download cheap (key for the consent flow below). Revisit B only if guaranteed first-visit offline becomes a requirement.

### 2. Tooling
Use `vite-plugin-pwa` in **`injectManifest`** mode (hand-written SW with Workbox helpers) over `generateSW` — we need custom update gating, COOP/COEP-safe responses, and runtime strategies. `manifest.webmanifest` generated by the plugin.

## What to cache (and what NOT to)

Derive the candidate list from Vite's build manifest, but **classify** rather than precache-all (dist is ~124MB, mostly sourcemaps + Monaco):

| Bucket | Examples | Strategy |
|---|---|---|
| App shell (needed to boot offline) | `index.html`, main/entry chunks, css, fonts, `processors.js`, `workers-main.js`, `offline-engine.js` | `CacheFirst` (immutable uuid) — the **only** precache, if any |
| Heavy lazy libs | Monaco, FFmpeg WASM, onnxruntime | `CacheFirst`, cache-on-first-use, max-entries cap. **Never precache** |
| `build-info.json`, `index.html` | version probe | `NetworkFirst` / network-only — never serve stale (would break update detection) |
| Cloud assets | `assets.opendaw.studio`, `api.opendaw.studio` | **NetworkOnly** (no opaque caching; CORP/COEP issues) — failures drive offline UI |
| Sourcemaps, `.map`, source `.ts` | — | **Exclude entirely** |

**File-selection method:** add a build step that reads Vite's `.vite/manifest.json`, filters to shell entrypoints + their static imports (exclude `.map`, exclude dynamic-import-only chunks like monaco/ffmpeg/onnx), and emits a `precache.json`. The SW imports that. This is how we *know* which files matter instead of globbing blindly.

**Brotli gotcha:** SW Cache API stores whatever it fetches. With Apache rewriting to `.br` + `Content-Encoding: br`, the cached `Response` keeps that header and the browser decompresses transparently — OK, but only if we fetch the logical URL (`x.js`, not `x.js.br`) and let negotiation happen. Do **not** add `.br` URLs to the cache list. Verify decompression of cached worklet/wasm responses early.

## Update flow (consent-gated download) — the crucial part

Requirement: when online and a newer build exists, **ask before downloading**, then update cleanly without data loss (OPFS is safe; in-memory project is not).

Because option A keeps the install download tiny, "download" really means *activate the new shell + fetch new uuid bundles*. Flow:

1. **Detect** (reuse existing poll): `build-info.json` uuid != running `BUILD_UUID` → new version online. Keep the 5s/`navigator.onLine` guard.
2. **Disable silent SW auto-update**: register with `registerType: 'prompt'` and **do not** call `registration.update()` on a timer or rely on the browser's background update to pull bundles. (Background SW byte-check still happens, but with option A the SW byte content rarely changes, so no surprise heavy download.)
3. **Prompt** the user (replace the passive `UpdateMessage` banner with a dialog): "A new version is available. Update now? Unsaved changes will be lost — save first." Offer *Update now* / *Later*.
4. **On consent only**: call `registration.update()` → new SW installs/precaches shell → on `waiting`, post `SKIP_WAITING` → on `controllerchange`, `location.reload()`. This is the point where bytes are fetched, so consent genuinely gates the download.
5. **On decline**: keep running the old (fully cached) version; re-offer on next detection or next launch.
6. **Cache-mismatch guard** (`boot.ts:132`): with a SW present this should no longer fire from stale HTML; keep it as a backstop but route to the same update prompt instead of a dead-end `Dialogs.cache()`.

Edge cases to handle: multiple tabs (one updates, others get `controllerchange` → warn/reload); update offered mid-recording (defer, never auto-reload); failed update fetch (stay on current version, surface error).

## Offline UI degradation

Add a single source of truth for connectivity and branch the UI off it:

- **Signal**: combine `navigator.onLine` with actual fetch outcomes to `api.opendaw.studio` (onLine lies — true on captive/LAN). Expose as an observable (`MutableObservableValue<boolean>` / lib-std) so views subscribe.
- **Degrade when offline**: sample & soundfont browsers → show "Library requires connection" + list only OPFS-cached items; cloud login (Dropbox/GDrive), publish-music, stats/latency/sponsors/GitHub panels → disabled/hidden with an offline notice; suppress update polling and visitor/latency reporters.
- **Stays available offline**: editing, OPFS projects/samples/presets, worklet DSP, offline-engine render — all local. Make the offline boundary explicit so the app never hangs on a dead `fetch`.
- Wrap every cloud `fetch` in `tryCatch`/`Promises.tryCatch` and a timeout so offline = fast graceful fallback, not a spinner.

## Manifest + install

- `manifest.webmanifest`: name, `display: standalone`, theme/background, icons (reuse `public/images`, add 192/512 maskable PNGs). `start_url: "/"`, `scope: "/"`.
- Keep COOP/COEP meta + headers; confirm `standalone` launch is still `crossOriginIsolated` (it is, headers come from the server/SW response, not the launch context — verify).
- Desktop-only: gate the install prompt the same way `main.ts` gates mobile.

## Deployment changes

- Ship `sw.js` + `manifest.webmanifest` to **web root** (not the release dir) so scope is `/`. Add `.htaccess` rules: `sw.js` and `manifest.webmanifest` → `no-cache`; add `Service-Worker-Allowed: /` if any per-release SW variant is used.
- SW must serve cached responses **with COOP/COEP/CORP intact** — when synthesizing/replaying responses, preserve or re-add the three headers, or offline pages lose cross-origin isolation and the worklet/SharedArrayBuffer boot aborts. This is the single most likely thing to break. Test offline `crossOriginIsolated === true` before anything else.
- Add a post-build step to emit `precache.json` (see file-selection above) into the release dir.
- CI: no infra change (still SFTP); just two extra root files + the precache manifest.

## Risks / open questions

1. **COOP/COEP through the SW** — top risk; offline isolation must hold or the app won't boot. Prototype this first.
2. **Brotli + Cache API** — verify cached worklet/wasm decompress correctly.
3. **Background SW auto-update bypassing consent** — option A minimizes the SW's own byte-churn so this is mostly moot, but confirm the browser never silently pulls fresh bundles without the prompt.
4. **First-visit offline** — option A gives offline only after one online visit; acceptable? If not → option B.
5. **OPFS quota under heavier caching** — request `navigator.storage.persist()` (already partly done) and surface quota in offline UI.
6. **Stale `build-info.json` via SW** — must stay network-first or update detection silently dies.

## Phased rollout

1. **Spike**: root SW (option A), no precache, runtime cache the shell; prove offline boot with `crossOriginIsolated` + worklets + Brotli. (De-risks #1, #2.)
2. **Manifest + installability**, desktop-gated.
3. **Consent-gated update flow**, replacing the passive banner; multi-tab + mid-record edge cases.
4. **Offline UI degradation** via the connectivity observable.
5. **precache.json** build step + tune cache buckets/limits.
6. Ship to `dev.opendaw.studio` first; soak; then `main`.
