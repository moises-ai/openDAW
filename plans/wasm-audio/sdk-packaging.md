# SDK packaging & configurable plugin path

Make the WASM engine consumable by any SDK app the way the TS engine is today: `install(...)` with
self-hosted URLs, no studio-app code, no relative source imports. Complements [09-rollout](09-rollout.md)
(flag/exposure) — this is about **where the engine lives and how a host configures it**.

## STATUS: implemented (2026-07-06)

`packages/studio/core-wasm` = `@opendaw/studio-core-wasm`. dist ships `index.js` (main-thread API),
`wasm-processor.js` + `wasm-offline-worker.js` (prebuilt esbuild bundles, worklet-scope shim first) and
`wasm/engine.wasm` + `wasm/plugins/*.wasm` (built by the package's own `build-wasm.sh`, wasm-opt'ed).
`WasmEngine.install({processorUrl, offlineWorkerUrl, wasmUrl})` as planned. The studio consumes the
published surface (`?url` asset imports + the vite plugin copies `dist/wasm` under `/wasm-engine/`);
`app/wasm` stays the dev harness and deep-imports the package **src** (fast Rust loop, no dist staleness).
The SDK meta-package re-lists it; `studio/sdk/README.md` documents the boot. `test/dist-smoke.test.ts`
is the consumer smoke test (dist-only resolution, device-table-vs-built-plugins drift, scope-shim order,
engine compile). Deviations from the proposal: **no `plugins.json`** — the DEVICES/COMPOSITES table stays
compiled-in TS data (the smoke test guards drift against the built plugins); manifest-driven subsetting
can layer on later without breaking the API. Sine stays an app-wasm-only artifact (its slim
`build-wasm.sh` wraps the package's and adds sine).

## Current state (what blocks SDK use)

Shipped in the studio (dev channel): the header toggle, the `EngineVariant` seam in `studio-core`
(published ✓), and the studio-contract worklet processor. But:

- The integration lives in **private app packages**: the processor + main-thread wiring in
  `app/studio/src/wasm-engine/`, the shared plumbing (engine-modules, device-linker, bridges,
  engine-exports, serialize-update-tasks, worklet-scope shim, soundfont-simplify) in `app/wasm/src/`,
  imported by **relative source paths**. None of it is published.
- The artifact path is **hardcoded** (`${BASE_URL}wasm-engine`), served by a studio-only vite plugin
  copying from `app/wasm/public/`.
- The device table (`DEVICES` in engine-modules.ts) is a **fixed compiled-in list** of
  `{url, boxType}` — a host cannot relocate, subset, or extend it.
- The wasm binaries are produced by `build-wasm.sh` into `app/wasm/public/` and are not part of any
  npm artifact.

## Decisions (proposed)

### 1. A published engine package: `@opendaw/studio-core-wasm`

Sibling to `studio-core-workers` / `studio-core-processors`, versioned on the same release train
(the binaries embed the box-schema WASM CONTRACT — lockstep with `studio-boxes`/`studio-adapters`).

- **dist ships the built artifacts**: `engine.wasm`, all `device_*.wasm`, a generated
  `plugins.json` manifest, and a **prebuilt worklet module** (`wasm-processor.js`, bundled like
  `processors.js` so hosts need neither our vite worker pipeline nor knowledge of the
  `self`/`location` worklet-scope shim).
- **dist ships the main-thread API** (see 2).
- **src absorbs the shared plumbing** from `app/wasm/src/` (engine-modules, engine-exports,
  device-linker, script-bridge, nam-bridge, soundfont-simplify, serialize-update-tasks,
  worklet-scope). `app/wasm` and the studio then import from the package — one source of truth,
  the test app keeps its fast Rust iteration loop (dev override below).
- **build**: turbo task runs `build-wasm.sh` (Rust toolchain — already in deploy/parity CI; the
  publish workflow gets the same block), bundles the worklet entry, emits `plugins.json`, copies
  binaries to dist. `wasm-opt` in CI (binaryen) for size.

### 2. Configurable plugin path — mirror the `install(url)` convention

```ts
WasmEngine.install({
    processorUrl,             // the prebuilt worklet module (copy from dist, host anywhere)
    pluginsUrl                // base URL serving engine.wasm + device_*.wasm + plugins.json
})
await WasmEngine.ensureReady(context)   // compile + addModule + EngineVariant provider
```

- `plugins.json` is **generated by `build-wasm.sh`** from `DEVICE_CRATES` + the box-type table:
  `[{file, boxType, kind}, ...]` + composites. The loader fetches the manifest, then each module
  from `pluginsUrl` — the compiled-in `DEVICES` list becomes data. A host can subset (smaller
  deploys) and later **extend** (third-party device plugins land here; the engine already treats
  devices as runtime-loaded side modules, see [device-plugins](device-plugins.md)).
- Engine selection stays host-owned: the SDK exposes `WasmEngine.setEnabled/isEnabled` +
  `EngineVariant` — the studio's header toggle remains app code.

### 3. Studio app consumes the package

- Delete the relative imports in `app/studio/src/wasm-engine/` (keep only the toggle UI + flag).
- The vite plugin copies dist artifacts from the package instead of `../wasm/public` — with a
  **dev override** pointing at `app/wasm/public/` so Rust iteration does not require a package
  rebuild.

### 4. SDK docs + consumer smoke test

- `studio/sdk/README.md` gains a boot snippet (workers/worklets/offline + `WasmEngine.install`),
  the COOP/COEP + SharedArrayBuffer requirement, and the current feature gaps (recording,
  monitoring, freeze, composite cells — see the studio-integration memory/rollout doc).
- A minimal consumer smoke test that boots **from dist artifacts only** (no monorepo source
  imports) and renders a quantum — catches packaging regressions (missing file in dist, a stray
  relative import, a broken manifest) at publish time.

## Steps

1. Scaffold `packages/studio/core-wasm` (package.json, tsconfig, build = build-wasm.sh + worklet
   bundle + manifest + copy). Turbo: inputs `$TURBO_ROOT$/crates/**`, outputs `dist/**`.
2. Move the shared modules out of `app/wasm/src/`; both apps import from the package.
3. Public API: `WasmEngine.install({processorUrl, pluginsUrl})` / `ensureReady` / `setEnabled`;
   manifest-driven `loadEngineModules`.
4. Studio app migrates (toggle stays); dev override for the Rust loop.
5. SDK README + example; consumer smoke test; publish dry-run.
6. Publish workflow gets the Rust toolchain block (mirror deploy.yml).

## Open questions

- Package name: `@opendaw/studio-core-wasm` vs `@opendaw/studio-engine-wasm`.
- npm payload (~1.5 MB unoptimized binaries; wasm-opt shrinks) — acceptable, or split a
  `-binaries` package later?
- Should `ensureReady` failures auto-clear the persisted flag inside the package (as the studio
  does today at boot), or is that host policy?
- Capability gating ([09-rollout](09-rollout.md) §2) — where does the supported-feature list live
  once hosts can subset the plugin manifest?
