# Werkstatt MIDI Input (#277)

**Doability:** ⭐⭐⭐☆☆ (3/5) — clear template to mirror (instrument note-event consumption), but Werkstatt is an audio-effect today and needs a new "receive notes while staying an audio effect" wiring plus a script-facing API design, and must land in both TS and WASM engines
**Type:** feature
**Scope:** medium

## What is asked
Let a Werkstatt script receive Note On/Off events (e.g. to drive an ADSR envelope that modulates a ring-mod frequency) alongside its existing audio in/out processing. The user supplied a working ring-mod script and an AI-suggested `block.midi`/`io.midi` shape keyed to sample-frame offsets within the block.

## Current behaviour / relevant code
Werkstatt is implemented (not just planned) as an audio-effect, with no note-event path:
- `WerkstattDeviceBoxAdapter` (`packages/studio/adapters/src/devices/audio-effects/WerkstattDeviceBoxAdapter.ts:16-17`): `readonly type = "audio-effect"`, `readonly accepts = "audio"`.
- `WerkstattDeviceProcessor extends AudioProcessor implements AudioEffectDeviceProcessor` (`packages/studio/core-processors/src/devices/audio-effects/WerkstattDeviceProcessor.ts:44`). Its `reset()` clears `this.eventInput` (line 161) but there is no `handleEvent()` override and no `NoteEventTarget` implementation — any event actually delivered to it today hits the base class's panic.
- `AudioProcessor` base (`packages/studio/core-processors/src/AudioProcessor.ts:8-59`): `process()` iterates `this.eventInput.get(index)` per block (lines 15-51), slices audio at each event's sample offset, calls `handleEvent(event)` for non-`update` events, then `processAudio(chunk)`. Base `handleEvent` (line 57) panics — subclasses that want events must override it.
- Note events are `NoteLifecycleEvent = Id<NoteEvent> | NoteCompleteEvent` (`packages/studio/core-processors/src/NoteEventSource.ts:9-38`): start = `{type:"note-event", position, duration, pitch, velocity, cent, id}`, stop = `{type:"note-complete-event", position, pitch, id}`.
- Real consumer template: `VaporisateurDeviceProcessor.handleEvent()` (`packages/studio/core-processors/src/devices/instruments/VaporisateurDeviceProcessor.ts:189-195`) — `NoteLifecycleEvent.isStart(event)` → start a voice, `isStop(event)` → stop it. It implements `NoteEventTarget`; `setNoteEventSource()` (line 172) wires it into `NoteEventInstrument` (`packages/studio/core-processors/src/NoteEventInstrument.ts:34-48`), which pulls `source.processNotes(p0,p1,flags)` and pushes each into `this.#receiver.eventInput.add(index, event)`.
- The chain assembly lives in `MidiDeviceChain.#wire()` (`packages/studio/core-processors/src/MidiDeviceChain.ts:76-96`): `NoteSequencer` (reads the Notes track's `NoteEventBox`/`NoteEventCollectionBox`) → each enabled midi-effect (`setNoteEventSource`) → the instrument at the end (`input.setNoteEventSource(source)`).
- Closest existing midi-effect template: `ArpeggioDeviceProcessor` (`packages/studio/core-processors/src/devices/midi-effects/ArpeggioDeviceProcessor.ts`) implements both `NoteEventSource` and `NoteEventTarget` (`setNoteEventSource` lines 60-64, `processNotes()` lines 66-96) and its adapter has `readonly accepts = "midi"` (`ArpeggioDeviceBoxAdapter.ts:24`). Werkstatt would need the consumer half only (`NoteEventTarget`, like Vaporisateur), not the re-emitting half, since it stays an audio effect.

## Plan
1. **Decide the routing model.** Werkstatt is an audio-effect sitting in an audio-unit's FX chain, but note events today flow through the separate `MidiDeviceChain` into the instrument at the head of that same audio-unit. Options:
   - (a) Give `WerkstattDeviceProcessor` a `NoteEventTarget`-like hookup that taps the *same* audio-unit's note stream (read-only sidecar), so a Werkstatt sitting after the instrument in the chain still "sees" the notes that drove that instrument this block. Closest to the user's use case (drive an envelope in lockstep with the instrument).
   - (b) Let Werkstatt itself become addressable as a `setNoteEventSource` target off the `MidiDeviceChain`, i.e. actually be inserted where a midi-effect would be, changing `accepts`. This changes what track types can host it and is a bigger schema/UI change.
   Given the user's example (ring-mod frequency driven by an ADSR keyed to note on/off of the same instrument), (a) is the natural fit and much smaller: Werkstatt subscribes to the AudioUnit's already-existing note stream rather than becoming a MIDI-effect device type.
2. **Extend `WerkstattDeviceProcessor`** to implement `NoteEventTarget`-equivalent consumption: override `handleEvent()` (mirroring `VaporisateurDeviceProcessor.handleEvent`) to buffer incoming `NoteLifecycleEvent`s per block instead of panicking.
3. **Design the script-facing shape.** The user's AI-suggested `block.midi`/`io.midi` (events tagged with sample-frame offset within the block) matches the existing `Block`-based per-chunk dispatch Werkstatt already uses (`process(io, block)` in `WerkstattDeviceProcessor.processAudio`, `packages/studio/core-processors/.../WerkstattDeviceProcessor.ts:175-201`) — the host already slices audio at event boundaries via `AudioProcessor.process()`, so exposing `block.notes` (array of `{type, pitch, velocity, id, frameOffset}` for the current chunk) is consistent with the existing per-chunk contract rather than needing a new dispatch mechanism. Update the `UserProcessor`/`Block` type contracts (`werkstatt-examples.ts`, `werkstatt-default.js`, the manual `werkstatt.md`) to document the new field.
4. **WASM mirror (frozen-contract rule).** `crates/stock-devices/device-werkstatt/src/lib.rs` implements the same processor in Rust for the WASM engine — per project convention (`WASM CONTRACT` comments, `project_wasm_frozen_contracts`), any change to the block/event shape delivered to Werkstatt scripts must be mirrored bit-for-bit in the Rust crate, including how it reads note events from its `AudioUnit`'s note stream (check `crates/engine-env` for the equivalent of `NoteEventInstrument`/`handleEvent`).
5. **Editor/UX**: `WerkstattDeviceEditor.tsx` (`packages/app/studio/src/ui/devices/audio-effects/`) likely needs no change if notes are purely a script-side input (no new UI control), but the manual page and starter examples should show the ADSR/ring-mod example from the issue.

## Risks / open questions
- Routing model (1) needs a maintainer decision: "listen to my own audio-unit's notes" vs "become an insertable midi-effect." The former is smaller but only works if Werkstatt sits in a chain that has an upstream instrument; behavior when there is no note source (a pure audio-effect chain with no instrument) needs a defined no-op.
- Must be designed to survive the two-engine parity requirement (TS + WASM); this roughly doubles the implementation surface for a device change (per `project_scriptable_devices` memory: Werkstatt is already parity-tested, this issue's change must extend, not break, `werkstatt-parity.test.ts`/`werkstatt-automation-parity.test.ts`).
- Sample-accurate note timing within a block (the "matched to sample-frame offset" part of the AI suggestion) is already how `AudioProcessor.process()` slices chunks at event boundaries — need to confirm this composes correctly when both audio-effect chunk-slicing (existing) and note events (new) want to split the same block, since today only one event stream (`this.eventInput`) drives the slicing and Werkstatt would newly rely on it.
