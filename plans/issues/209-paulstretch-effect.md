# Paulstretch Effect (#209)

**Doability:** ⭐⭐☆☆☆ (2/5) — TS FFT/windowing primitives are directly reusable, but Paulstretch's huge analysis windows and inherent processing latency don't fit openDAW's real-time chain-device model without an architectural decision, and the Rust FFT needs generalizing either way.
**Type:** feature
**Scope:** large

## What is asked
Integrate the Paulstretch algorithm: extreme time-stretching (10x-1000x+) that avoids the rhythmic/granular artifacts of normal time-stretch, by taking large FFT windows, randomizing per-bin phase while keeping magnitude, and overlap-adding at a much slower hop rate than the analysis rate. Used for ambient/drone sound design (as in Audacity's built-in Paulstretch).

## Current behaviour / relevant code
- Reusable FFT: `packages/lib/dsp/src/fft.ts` — `class FFT(n: int)`, arbitrary power-of-2 size, in-place radix-2 transform. Reusable windowing: `packages/lib/dsp/src/window.ts` — `Window.create(type, n)` (Hanning/Hamming/Blackman/Bartlett/BlackmanHarris). Both are already combined for spectral analysis in `packages/lib/dsp/src/AudioAnalyser.ts` — confirming the primitives compose correctly, though that class does forward analysis only (no phase manipulation or inverse resynthesis today).
- No existing "render effect"/offline destructive processing framework was found in the timeline UI (checked for reverse/normalize-style per-region destructive operations) — openDAW's effects model is exclusively real-time chain devices (`AudioEffectHost`), not offline/destructive region processing. This matters because classic Paulstretch (as in Audacity) is normally an **offline, non-real-time render** given the scale of stretch factors and window sizes involved.
- Rust/WASM FFT: `crates/dsp/src/analyser.rs` is fixed-size (`NUM_BINS=512`/`FFT_SIZE=1024`) and built for continuous spectrum-analysis broadcasting, not large-window one-shot resynthesis. No general-purpose FFT crate is present (`grep -rn "rustfft"` across `crates/*/Cargo.toml` is empty) — matches the same gap noted for #188's HQ mode.
- Related existing granular time-stretch: `packages/studio/core-processors/src/devices/instruments/Tape/TimeStretchSequencer.ts` (see #201's plan) — architecturally unrelated to Paulstretch (transient-aligned segment looping, not FFT phase randomization), but is the only other "extreme audio manipulation" feature in the codebase for UI/UX precedent (its editor lives under `packages/app/studio/src/ui/timeline/editors/audio/`).

## Plan
1. **Resolve the architecture question first** (see Risks below) — decide whether Paulstretch is a real-time chain device (processes a rolling window continuously, usable live but with large inherent latency equal to the window size) or an offline render applied to an audio region/clip (matches how Audacity and most DAWs implement it, avoids the real-time-device latency problem entirely, but requires a "bounce/render" code path that doesn't currently exist in openDAW's device-only effects model).
2. Core DSP module `packages/lib/dsp/src/paulstretch.ts`: for each large analysis window (e.g. 4096-16384 samples, user-configurable), apply a `Window` (Hanning), run `FFT`, randomize each bin's phase (uniform random 0-2π) while preserving magnitude, inverse FFT, and overlap-add at a hop size much smaller relative to the stretch factor than the window advance through the source — this is the actual Paulstretch technique and has no precedent in this codebase to mirror; implement from the published algorithm description (well documented, public domain reference implementation by Nasca Octavian Paul).
3. If implemented as a **chain device**: new schema `PaulstretchDeviceBox` (`stretchFactor`, `windowSize`), processor buffers enough input to run the FFT/overlap-add pipeline continuously, reporting its window size as processing latency through whatever latency-compensation mechanism the project has (same open question flagged in #188). If implemented as a **region/clip render effect**: this needs its own new "destructive render" plumbing, likely a bigger lift than the DSP itself, since no such mechanism exists today — recommend checking with the maintainer whether this class of feature (offline region processing) is in scope at all before proceeding.
4. Editor: parameters for stretch factor and window size; a preview/render trigger if implemented as an offline effect.
5. WASM mirror: generalize `crates/dsp/src/analyser.rs`'s FFT to arbitrary sizes (shared need with #188), then port the same phase-randomization/overlap-add logic to a new `crates/stock-devices/device-paulstretch` (or equivalent, depending on the architecture decision in step 1).

## Risks / open questions
- **This is the central open question and should block implementation until resolved with the maintainer**: openDAW has no offline/destructive region-processing feature today — every effect is a real-time chain device. Paulstretch's classic use (extreme stretch factors, large windows) fits an offline render much better than a real-time device (which would need to expose enormous latency, e.g. hundreds of milliseconds to seconds, live in the chain). Shipping it as a real-time device is possible but will feel unlike every reference implementation of Paulstretch users are familiar with.
- Directly overlaps #188's Rust FFT-generalization need — sequence these together so the arbitrary-size FFT work happens once.
- Phase randomization quality depends on window size and overlap ratio tuning — needs perceptual tuning/testing against Audacity's reference output, not just a literal port of the math.
- If shipped as a chain device with rolling windows, decide how "stretch factor" interacts with the timeline (does the effect change the perceived tempo/duration of a live signal, which is unusual for a chain effect, or does it operate only on a captured/frozen buffer) — this needs product-level clarification before schema design.
