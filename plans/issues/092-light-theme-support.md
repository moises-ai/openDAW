# Light Contrast Theme Support (#92)

**Doability:** ⭐⭐⭐⭐☆ (4/5) — the color system is already a single indirection point (one JS object driving all CSS vars), which is exactly the shape needed; main cost is designing/tuning an actual light palette, not plumbing.
**Type:** feature
**Scope:** medium

## What is asked
MVP: a way to switch between dark and light UI themes (button or in settings), default to the browser/OS theme preference, and persist the user's explicit choice in localStorage.

## Current behaviour / relevant code
Colors are already centralized behind one indirection layer, not scattered across SASS files — this is the key finding.

- `packages/studio/enums/src/Colors.ts` — a single `Colors` object with ~17 named `Color` (HSL) values (`white`, `blue`, `gray`, `background`, `panelBackground`, etc.), and `initializeColors(root)` which writes every entry as a CSS custom property (`--color-<kebab-name>`) onto whatever element it's given, converting `Color` → HSL string. There is currently exactly **one** palette (a dark one — e.g. `background: new Color(197, 8, 7)`, `panelBackground: new Color(197, 14, 9)`).
- `packages/app/studio/src/main.ts:17` — calls `initializeColors(document.documentElement)` once at boot.
- `packages/app/studio/src/ui/surface/Surface.tsx:259` — calls `initializeColors(target.documentElement)` again for popout windows (multi-window support) — **any theme mechanism must re-apply to popouts too**, this call site is the tell.
- `packages/app/studio/src/colors.sass:1-3` — `:root { color-scheme: dark; --color: var(--color-gray); ... }` — `color-scheme` is hardcoded to `dark`, which also drives native form-control rendering (scrollbars, checkboxes) independent of the custom properties; this needs to flip with the theme too.
- All component SASS files consume the CSS vars (`var(--color-panel-background)`, etc.) — never hardcoded hex — so once the vars have a light-mode value, the visual swap is essentially free at the component level.
- **Existing settings infrastructure to reuse**: `packages/studio/core/src/StudioSettings.ts` is a `zod` schema (`StudioSettingsSchema`) with sections like `"editing": { "overlapping-regions-behaviour": z.enum(OverlappingRegionsBehaviourOptions), ... }`. `packages/studio/core/src/StudioPreferences.ts` wraps it: `Preferences.host("preferences", StudioSettingsSchema)` (from `@opendaw/lib-fusion`), which already persists to localStorage and is consumed reactively via `StudioPreferences.catchupAndSubscribe(...)` (see `packages/app/studio/src/service/StudioService.ts` `#listenPreferences()`, which already does exactly this pattern for other boolean toggles like `document.body.classList.toggle(...)`). The Preferences UI page (`packages/app/studio/src/ui/pages/PreferencesPage.tsx`) auto-renders a control for every schema entry via `PreferencePanel` + `PreferencesPageLabels.ts` — adding a schema field gets a working settings-UI toggle for free.

## Plan
1. **Add a light palette.** In `packages/studio/enums/src/Colors.ts`, either add a second `LightColors` object (adjusted lightness/contrast per named color, keeping the same accent hues so the brand still reads as openDAW) or restructure `Colors` into `{dark: {...}, light: {...}}` keyed sets. Change `initializeColors(root, palette)` to take the palette to apply, defaulting to dark for backward compatibility of the call sites.
2. **Add the setting.** In `StudioSettingsSchema` (`packages/studio/core/src/StudioSettings.ts`), add a new section, e.g. `"appearance": z.object({"theme": z.enum(["system", "dark", "light"])}).default({"theme": "system"})`, mirroring the existing `OverlappingRegionsBehaviourOptions` enum pattern used for `editing.overlapping-regions-behaviour`. This gets localStorage persistence and a Preferences-page UI control for free via the existing `PreferencePanel` machinery — add labels in `PreferencesPageLabels.ts` to match.
3. **Resolve "system" to an actual theme**: read `window.matchMedia("(prefers-color-scheme: light)")` (initial value + subscribe to `change` for live OS-theme switching while the app is open, matching the "css default based on browser theme" requirement) only when the setting is `"system"`; otherwise use the explicit user choice.
4. **Apply on boot and on change.** In `StudioService`'s `#listenPreferences()` (`packages/app/studio/src/service/StudioService.ts`), add a `StudioPreferences.catchupAndSubscribe(...)` entry for `"appearance"`/`"theme"` that: calls `initializeColors(document.documentElement, resolvedPalette)`, sets `document.documentElement.style.colorScheme` (or updates `colors.sass`'s hardcoded `color-scheme: dark` to a var-driven value) to match, and toggles a `theme-light`/`theme-dark` class on `document.documentElement` for any component that needs a hard branch rather than a var (e.g. syntax highlighting in the Monaco editor, if it hardcodes a dark theme).
5. **Re-apply to popout windows**: `Surface.tsx:259`'s existing `initializeColors(target.documentElement)` call must pass the same resolved palette so popped-out panels (mixer, piano roll, etc., wherever popouts are used) don't stay dark when the main window is light.
6. **Add a quick-toggle button** (the issue allows either a button or settings-only) somewhere in the persistent chrome (e.g. `Footer.tsx`, which already reads `StudioPreferences` for other toggles) for fast switching without opening the full Preferences page, in addition to the Preferences-page control from step 2.

## Risks / open questions
1. **Palette design is the real cost, not plumbing.** Every SASS file assumes dark backgrounds with light text and some hardcoded opacity tricks tuned for dark (`colors.sass`'s `$BRIGHTENER`/`$DARKENER` rgba overlays, `panel-background-stripes-dark`/`-bright` mixins) — these need visual QA against a light palette, not just a mechanical value flip. Treat this as a design pass, budget accordingly.
2. **Non-var color usage**: audit for any component that reads `Colors.gray` etc. directly in TS (e.g. `ColorsPage.tsx` renders `Object.entries(Colors)` directly) rather than through the CSS var — those call sites need the resolved-palette value threaded through instead of a static import, or they'll stay dark-only.
3. **Monaco editor / other embedded editors** likely have their own hardcoded dark theme config, independent of openDAW's CSS vars — needs a separate light-theme config passed to Monaco's `editor.defineTheme`/`setTheme` API, out of scope for the CSS-var mechanism alone.
4. **Region/track colors** (`ColorMenu.ts`, user-assigned track colors) are a separate concern from UI chrome theming — confirm they're unaffected (they should be, since they're user data, not theme-driven).
