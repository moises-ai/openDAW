# Classic time-stretch mode (#201)

**Doability:** ⭐⭐⭐☆☆ (3/5) — extends a well-structured existing voice-per-transient architecture with one new voice type, but decoupling "repeat rate" from transient-boundary timing is a real new timing model, not a copy-paste.
**Type:** feature
**Scope:** medium

## What is asked
Keep the existing time-stretch controls, and add a "classic" mode where, instead of looping the same segment on every repeat, each successive repeat advances the read position forward (through the source audio) so the repeated transient progresses rather than freezing on one loop point. Also add a way to speed up/slow down how often repeats happen, both tempo-synced and free-running in Hz.

## Current behaviour / relevant code
- The existing time-stretch is a per-region feature (not a chain device): `packages/studio/boxes/src/AudioTimeStretchBox.ts:30-34` — fields `warpMarkers` (pointer), `transientPlayMode` (int32, default `2`), `playbackRate` (float32, 0.5-2.0). Enum: `packages/studio/enums/src/TransientPlayMode.ts:1` — `enum TransientPlayMode { Once, Repeat, Pingpong }`.
- Adapter: `packages/studio/adapters/src/audio/AudioTimeStretchBoxAdapter.ts:49-59` exposes `playbackRate` and `transientPlayMode`.
- Core algorithm: `packages/studio/core-processors/src/devices/instruments/Tape/TimeStretchSequencer.ts` — a transient-aligned granular sequencer. On each block it locates the current transient segment via `EventCollection.floorLastIndex`, and at each transient boundary (`#handleTransientBoundary`) spawns a new `Voice` reading `[startSamples, endSamples)` at `effectivePlaybackRate`. Three voice types, one file each: `OnceVoice.ts` (play once, stop), `RepeatVoice.ts` (loops `[start+margin, end-margin]` with a crossfade — this is the "sustained segment loops in place, re-triggering the same audio" behavior the issue wants replaced by "classic" mode), `PingpongVoice.ts` (bounces forward/back with an equal-power crossfade). Dispatch is in `#createVoice`.
- Rust mirror: `crates/engine/src/time_stretch.rs` — line-for-line port, same `TransientPlayMode`/`Fade`/`Voice` enum (`OnceVoice`/`RepeatVoice`/`PingpongVoice`), `RepeatVoice::process` (lines 308-347) is the equivalent loop point. `TimeStretchConfig` (lines 54-58) holds `playback_rate`; a new repeat-rate/sync field extends this struct.
- UI: `packages/app/studio/src/ui/timeline/editors/audio/TimeStretchEditor.tsx`, `StretchSelector.tsx`, `TransientMarkerEditor.tsx`, `WarpMarkerEditor.tsx`.
- Sync-vs-Hz coexistence precedent, two flavors: (a) toggle-based — `packages/studio/boxes/src/VaporisateurLFO.ts` has a `sync` `BooleanField` (field 3) alongside a `rate` Float32Field (field 2, Hz, min 0.0001) that reinterprets the same rate field as tempo-synced when true; (b) additive — `DelayDeviceProcessor.ts:108-128` sums a tempo-synced term (`PPQN.pulsesToSamples(Fraction.toPPQN(fraction), bpm, sampleRate)`) and a free-running term (`parameterMillisTime.getValue() * 0.001 * sampleRate`) into one offset, with the free-running field (`DelayDeviceBox.ts`'s `lfoSpeed`, plain Hz 0.1-5 exponential) having no sync counterpart of its own.

## Plan
1. Extend `TransientPlayMode` enum (`packages/studio/enums/src/TransientPlayMode.ts`) with a new variant, e.g. `Classic` (or `Advance`) — must be added in lockstep in the Rust mirror's enum in `crates/engine/src/time_stretch.rs` (frozen-contract rule: enum ordinals must match).
2. Add new fields to `AudioTimeStretchBox.ts`: a `repeatRate` (float32, Hz, exponential, mirroring `DelayDeviceBox`'s `lfoSpeed` shape) and a `repeatSync` boolean (mirroring `VaporisateurLFO`'s `sync` field) plus, if synced, a fraction/PPQN field for the synced rate (mirroring `DelayDeviceBox`'s synced-fraction field feeding `DelayDeviceProcessor.ts:108-128`'s `sync` term). Update `AudioTimeStretchBoxAdapter.ts` to expose all three.
3. New voice type `ClassicVoice.ts` (TS) alongside `OnceVoice`/`RepeatVoice`/`PingpongVoice`: instead of looping the same `[start+margin, end-margin]` window in place (as `RepeatVoice` does), on each repeat cycle it advances its read window forward through the source by one step (e.g. re-derives `[start, end)` from the *next* position in the source at the configured repeat rate, rather than waiting for the next natural transient boundary) — crossfading between the old and new read positions the same way `RepeatVoice` crossfades loop points, so the timing driving retriggers is the new `repeatRate`/`repeatSync` parameter instead of (or in addition to) transient detection.
4. Wire `#createVoice`'s dispatch in `TimeStretchSequencer.ts` to construct a `ClassicVoice` when `transientPlayMode === TransientPlayMode.Classic`, passing the resolved repeat-rate (computed the same additive or toggle way as the sync precedents above) as the voice's retrigger interval.
5. Port the same `ClassicVoice` logic to `crates/engine/src/time_stretch.rs`, extending the `Voice` enum and `TimeStretchConfig` with the new rate/sync fields, matching the TS float math exactly (per the project's "mod_euclid bit-exact TS mirror" experience with time-stretch divergence — floating point order must match).
6. UI: add repeat-rate (Hz) and sync toggle/fraction controls to `TimeStretchEditor.tsx`/`StretchSelector.tsx`, and add "Classic" to whatever mode selector currently offers Once/Repeat/Pingpong.

## Risks / open questions
- Precise definition of "advances forward" needs nailing down with the maintainer: does each repeat jump to the *next transient's* segment (transient-driven advance) or advance by a fixed sample/time step independent of transient positions (rate-driven advance, matching "increase/decrease the repeat rate" framing)? These are different algorithms — the issue's phrasing ("after every repeat the repeated transient advances forward" + "a way to increase/decrease the repeat rate") suggests the latter (rate-driven), decoupling retrigger timing from transient detection entirely for this mode.
- This module already has a documented history of TS/Rust floating-point divergence causing audible artifacts (project memory: "Werkstatt stutter divergence" — `mod_euclid` vs TS `fract`-based mod mismatch cost ~10dB in a real bug). Any new modulo/wrap-around math in `ClassicVoice` must mirror the TS operation order exactly in Rust from the start, not be fixed after the fact.
- Relates to the "Time-stretch v2" plan already on file (project memory: transient-aware phase vocoder plan, written but not implemented) — confirm this "classic" mode request is independent, incremental work on the current granular engine, not blocked on or superseded by that larger rewrite.
