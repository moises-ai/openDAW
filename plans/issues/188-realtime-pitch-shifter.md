# Real-time Audio Pitch Shifter (#188)

**Doability:** ⭐⭐☆☆☆ (2/5) — no existing audio pitch-shift DSP anywhere in the repo (the only "pitch" device is MIDI note transposition); both requested modes are non-trivial new algorithms needed cross-engine, plus a licensing question on the suggested reference implementation.
**Type:** feature
**Scope:** large

## What is asked
A chain-device pitch shifter with two modes: **Live** (low latency, time-domain granular/dual-delay technique) and **HQ** (phase vocoder or WSOLA, higher quality, more latency). Issue references SoundTouchJS (SOLA algorithm, LGPL-licensed) as a possible portable reference.

## Current behaviour / relevant code
- **No audio pitch-shifting DSP exists.** `crates/stock-devices/device-pitch/src/lib.rs` is a MIDI effect only (shifts note-on/off pitch by octaves/semitones/cents) — unrelated to audio-domain pitch shifting. `packages/studio/scripting/src/impl/pitcheffectimpl.ts` is the same MIDI concept. `grep -ri "wsola\|sola\|granular"` across `packages/lib` and `crates` returns no audio pitch-shift hits (the only "granular" behavior in the repo is the time-stretch sequencer, which changes duration, not pitch, at a fixed sample rate — see #201's plan).
- Existing FFT/window infra usable for **HQ mode** (phase vocoder): TS `packages/lib/dsp/src/fft.ts` (`class FFT`, arbitrary power-of-2 size `n`, in-place radix-2) + `packages/lib/dsp/src/window.ts` (`Window.create(type, n)`, Hanning/Hamming/Blackman/etc.), already combined in `packages/lib/dsp/src/AudioAnalyser.ts` for spectral analysis (not currently used for resynthesis/pitch-shifting, but the primitives are directly reusable for an overlap-add phase vocoder: FFT → phase-adjust per bin using a pitch-shift ratio → inverse FFT → overlap-add).
- Existing fractional delay-line infra usable for **Live mode** (time-domain granular/dual-delay pitch shift): `packages/lib/dsp/src/delay.ts` (`class Delay`, interpolated ring buffer with smooth offset ramping) — the standard "two crossfaded delay taps with a sawtooth-modulated read rate" pitch-shift technique builds directly on this, same primitive proposed for chorus (#195).
- Rust/WASM side has **no general-purpose FFT** available for HQ mode: `crates/dsp/src/analyser.rs` is a hand-ported, allocation-free radix-2 FFT fixed at `NUM_BINS=512`/`FFT_SIZE=1024` for spectrum analysis — not structured for the larger, variable window sizes a quality phase vocoder needs, and `grep -rn "rustfft"` across all `Cargo.toml` in `crates/` returns nothing (no such dependency exists). Either generalize `analyser.rs`'s FFT to arbitrary sizes, or add a new dependency — the project's stated preference is minimal external dependencies, so generalizing the existing radix-2 implementation is the better fit.

## Plan
1. **Live mode** (time-domain, low latency): implement a classic dual-delay-line pitch shifter — two `Delay` instances reading at a rate offset from 1.0 (proportional to the pitch-shift ratio), each with an independent sawtooth-driven read-position ramp, crossfaded (equal-power) whenever either delay line's read position wraps, to avoid clicks (mirror the crossfade idiom already used in `RepeatVoice.ts`/`PingpongVoice.ts` for the same purpose). New processor file, new box schema (`ratio`/semitones, `mode` toggle Live/HQ, `mix`).
2. **HQ mode** (phase vocoder): implement analysis/resynthesis using the existing `FFT`+`Window` classes — overlap-add framing, per-bin phase unwrapping and phase-increment scaling by the pitch ratio, inverse FFT, overlap-add reconstruction. This is materially more code than Live mode and has no existing precedent to mirror in this repo; budget it as the bulk of the implementation effort. Consider whether this can share code with a future phase-vocoder-based time-stretch (project memory: "Time-stretch v2" plan already proposes a phase vocoder, written but not implemented) — if both land, factor the shared phase-vocoder core (STFT framing, phase accumulation) into one `@opendaw/lib-dsp` module reused by both features rather than duplicating it.
3. Schema: new `PitchShiftDeviceBox` (name TBD to avoid clashing with the existing MIDI `PitchDeviceBox`), fields: `mode` (int32, Live/HQ), `pitch` (float32, semitones or ratio, bipolar), `mix` (unipolar %), `latency`/`quality`-related fields as needed per mode.
4. Adapter, processor, editor following the standard four-layer pattern (see `plans/waveshaper-device.md` for the registration checklist: `EffectFactories.ts`, `DeviceProcessorFactory.ts`, `BoxAdapters.ts`, `DeviceEditorFactory.tsx`, `DeviceManualUrls.ts`).
5. WASM mirror: new crate `crates/stock-devices/device-pitch-shift` (name TBD). Live mode ports directly (same `Delay`-equivalent primitive already used by `device-delay`). HQ mode requires generalizing `crates/dsp/src/analyser.rs`'s FFT to arbitrary window sizes first (or introducing a minimal, from-scratch radix-2 implementation parameterized by size, matching the TS `FFT` class's approach bit-for-bit per the project's cross-engine parity rules).

## Risks / open questions
- **Licensing**: SoundTouchJS is LGPL. The project's stated preference is minimal external dependencies and homebrew implementations — recommend implementing the SOLA/phase-vocoder algorithm from first principles (well-documented, standard DSP techniques) rather than vendoring or porting LGPL-licensed source, to avoid license obligations on a codebase that otherwise has none. Flag this explicitly for the maintainer's decision before any code lands.
- HQ mode's latency (an FFT-window's worth of samples, typically tens of milliseconds) needs to be reported through whatever latency-compensation mechanism openDAW already has for other latent devices (e.g. lookahead compressor, see #79) — confirm that mechanism exists and is reusable, otherwise HQ mode will desync in the mix.
- Real scope risk: this is effectively two full DSP algorithms (SOLA-style granular + phase vocoder) implemented twice (TS + Rust) — likely the largest single ask in this batch. Consider shipping Live mode alone first as a smaller, faster win, with HQ mode as a clearly separated follow-up.
- Overlaps with #201 (classic time-stretch) and #209 (Paulstretch) in FFT/phase-vocoder infrastructure needs — worth sequencing these three together so the FFT-generalization work (TS windowing already fine; Rust FFT needs arbitrary-size support) is done once, not three times.
